# ทำไม 1 ไฟล์ `.lance` ถึงเก็บข้อมูลได้หลายแถว และทำไมบางข้อมูลอยู่ไฟล์เดียวกัน บางอันแยกไฟล์

> อ้างอิงจากโครงสร้างจริงใน `~/.oracle/lancedb/oracle_knowledge.lance`

## `.lance` จริง ๆ แล้วเป็น "โฟลเดอร์" ไม่ใช่ไฟล์เดียว

`oracle_knowledge.lance` ไม่ใช่ไฟล์ แต่เป็น **โฟลเดอร์** ที่มี 3 ส่วน:

- `data/` — ข้อมูลแถวจริง ถูกแบ่งเก็บเป็น **fragment files** (ไฟล์ `*.lance` ที่เห็นข้างใน)
- `_versions/*.manifest` — snapshot ของตารางในแต่ละเวอร์ชัน
- `_transactions/*.txn` — log ของทุก ๆ การเขียน (write operation)

## ทำไม 1 fragment เก็บได้หลายแถว

ไฟล์ fragment แต่ละไฟล์เก็บข้อมูลแบบ **columnar** (รูปแบบ Arrow/Lance) คือเก็บเป็น
**ก้อน (batch) ของหลายแถว** ไม่ใช่แถวเดียว

เวลา Oracle เขียนข้อมูลทีเดียว เช่น knowledge 30 รายการ ทั้ง 30 แถวจะถูก serialize
ลงไปใน fragment file **เดียว** ดังนั้น "1 ไฟล์ = หลายแถว" จึงเป็นเรื่องปกติ —
fragment คือ "ชิ้นส่วนของตาราง" ไม่ใช่ "1 เรคคอร์ด"

นี่คือเหตุผลที่ขนาดไฟล์ใน `data/` ต่างกันมาก:
- ไฟล์ `~100KB` = การเขียนที่ batch หลายแถว / แถวใหญ่
- ไฟล์ `4KB`, `6KB`, `19KB` = การเขียนที่เพิ่มแค่ไม่กี่แถว

## ทำไมบางแถวอยู่ไฟล์เดียวกัน บางแถวแยกไฟล์

ไม่ได้ขึ้นกับ *เนื้อหา* ของข้อมูล แต่ขึ้นกับว่า **มาจากการเขียน (write) ครั้งไหน**

Lance fragment เป็นแบบ **immutable (เปลี่ยนแก้ไม่ได้)**: เมื่อเขียน fragment ไปแล้ว
Lance จะ **ไม่** เปิดไฟล์เดิมขึ้นมาเขียนเพิ่ม ทุกครั้งที่ insert/add จะสร้าง
**ไฟล์ fragment ใหม่เสมอ**

ดังนั้น:
- แถวที่ถูกเขียน **พร้อมกันใน transaction เดียว** → อยู่ไฟล์ fragment **เดียวกัน**
- แถวที่เขียน **คนละ transaction** → อยู่ไฟล์ fragment **แยกกัน**

ดูได้จากข้อมูลจริง: `_transactions/` มี entry เลข `0` ถึง `19` (20 transactions) และ
`data/` มี fragment 19 ไฟล์ — transaction `0` คือการสร้างตาราง ที่เหลือส่วนใหญ่
แต่ละครั้งสร้าง 1 fragment ส่วน timestamp ก็ตรงกัน เช่น กลุ่มไฟล์ช่วง `07:21`–`07:22`
ก็ตรงกับ transaction `14`–`19` ที่เขียนในช่วงเวลานั้น

## สรุปแบบใช้งานจริง

ดีไซน์แบบ append-only fragments นี้คือเหตุผลที่ knowledge base ซึ่งเขียนทีละนิด
(ไม่กี่รายการต่อ session) จะกลายเป็นไฟล์เล็ก ๆ หลายไฟล์ แทนที่จะเป็นไฟล์ใหญ่ไฟล์เดียว

LanceDB มีคำสั่ง maintenance ชื่อ **compaction** (`optimize` / `compact_files`) ที่จะ
รวม fragment เล็ก ๆ หลายไฟล์ให้เป็นไฟล์ใหญ่ไม่กี่ไฟล์ เพื่อให้อ่านเร็วขึ้น ถ้ารันคำสั่งนี้
19 ไฟล์จะยุบเหลือ 1–2 ไฟล์ และความสัมพันธ์ "แถว → ไฟล์" ก็จะไม่ตรงกับขอบเขตการเขียน
เดิมอีกต่อไป

**สั้น ๆ:**
- **หลายแถวต่อไฟล์** → fragment เก็บทั้ง batch จากการเขียน 1 ครั้ง
- **ไฟล์เดียวกัน vs คนละไฟล์** → มาจาก transaction เดียวกัน หรือคนละ transaction
  เพราะ fragment เปลี่ยนแก้ไม่ได้ ทุกการเขียนจึงสร้างไฟล์ใหม่
