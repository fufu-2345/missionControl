# Commit vs ไม่ Commit — เมื่อไหร่ progress จะหาย?

> เขียนเมื่อ 2026-06-21 — ตอบ 3 คำถาม: (1) ถ้าไม่ commit เมื่อไหร่ progress จะหาย, (2) commit กับไม่ commit ต่างกันยังไง, (3) switch branch เกิดเมื่อไหร่

---

## สรุปสั้น (TL;DR)

- **ไม่ commit = ไฟล์ไม่หาย** ไฟล์ถูกเซฟลงดิสก์ทันทีที่แก้ การไม่ commit ไม่ได้ทำให้ไฟล์หายไป
- progress จะหาย **เฉพาะตอนที่รันคำสั่ง git ที่ "ทิ้ง" งาน** (เช่น `reset --hard`, `checkout .`, `clean -fd`) หรือลบไฟล์เอง หรือดิสก์พัง
- **switch branch ปกติไม่ทำให้งานหาย** — git จะกั้นไว้ ถ้างานที่ยังไม่ commit จะถูกทับ มันจะ error และไม่ยอมสลับให้

---

## 1. แนวคิดหลัก: Save ≠ Commit

นี่คือจุดที่คนสับสนมากที่สุด **การเซฟไฟล์ กับ การ commit เป็นคนละเรื่องกัน**

| สิ่งที่ทำ | ความหมาย | งานหายไหมถ้าข้ามขั้นนี้? |
|---|---|---|
| **เซฟไฟล์ (save)** | ไฟล์ถูกเขียนลงดิสก์ในเครื่อง | ไฟล์ถูกเซฟอยู่แล้วเสมอ |
| **Commit (local)** | ถ่าย *snapshot / จุดเซฟ* ของไฟล์เก็บไว้ใน git history | **ไม่หาย** — ไฟล์ยังอยู่บนดิสก์ |
| **Push** | ส่ง commit ขึ้น GitHub (สำรองข้อมูลออกนอกเครื่อง) | กันเครื่องพัง / ดิสก์เสีย |

**เปรียบเทียบกับเกม:**
- **เซฟไฟล์** = เกมกำลังเล่นอยู่ ตัวละครอยู่ตรงนี้
- **Commit** = สร้าง "เซฟพอยต์" ที่ย้อนกลับมาได้ทีหลัง
- **Push** = อัปเซฟขึ้น cloud กันเครื่องพัง

ถ้าไม่สร้างเซฟพอยต์ ตัวละครไม่ได้หายไปไหน — แค่ "ย้อนเวลา" กลับไปจุดเก่าไม่ได้เท่านั้น

---

## 2. ความต่างระหว่าง Commit กับ ไม่ Commit

| หัวข้อ | **ไม่ commit** (uncommitted) | **Commit แล้ว** (local commit) |
|---|---|---|
| ไฟล์อยู่บนดิสก์ไหม | อยู่ ✅ | อยู่ ✅ |
| มีประวัติ / ย้อนกลับได้ไหม | ❌ ย้อนไม่ได้ | ✅ ย้อนได้ |
| ปลอดภัยจากคำสั่ง git อันตราย | ❌ เสี่ยงโดนทิ้ง | ✅ ปลอดภัย (อยู่ใน history แล้ว) |
| สลับ branch ได้สะดวกไหม | บางทีติด ต้อง stash/commit ก่อน | ✅ สลับได้เลย |
| สำรองขึ้น GitHub ได้ไหม | ❌ ต้อง commit ก่อน push | ต้อง `push` อีกที ถึงจะขึ้น cloud |

**สรุป:** commit (local) ไม่ได้แปลว่าขึ้น GitHub แล้ว — มันแค่เก็บจุดเซฟไว้ "ในเครื่อง" ถ้าอยากกันเครื่องพังต้อง `git push` ต่อ

---

## 3. เมื่อไหร่ progress จะหาย ถ้าไม่ commit? ⚠️

การ "ไม่ commit เฉยๆ" ปลอดภัย — ปิดเครื่อง พักงาน วันรุ่งขึ้นมาเปิดไฟล์ก็ยังอยู่ครบ

งานที่ยังไม่ commit จะหาย **เฉพาะเหตุการณ์เหล่านี้:**

### 3.1 รันคำสั่ง git ที่ "ทิ้ง" การแก้ไข
```bash
git reset --hard          # ลบการแก้ทั้งหมด กลับไป commit ล่าสุด → งานที่ยังไม่ commit หาย
git checkout .            # คืนไฟล์ทั้งหมดเป็นเวอร์ชัน commit ล่าสุด → หาย
git restore .             # เหมือน checkout . (คำสั่งใหม่) → หาย
git clean -fd             # ลบไฟล์ใหม่ที่ยังไม่ถูก track ทิ้ง → หาย
git stash                 # เก็บงานซ่อนไว้ (ไม่หายทันที แต่ถ้า drop ทิ้งก็หาย)
```
> คำสั่งที่มี `--hard`, `clean`, `checkout .`, `restore .` = ระวังเสมอ ถ้ายังไม่ commit

### 3.2 ลบไฟล์ / โฟลเดอร์เอง
ลบไฟล์ที่ยังไม่ commit = git ช่วยกู้ไม่ได้ (เพราะไม่เคยมี snapshot)

### 3.3 ดิสก์พัง / เครื่องพัง
git ในเครื่อง **ไม่ใช่ backup** ถ้าไม่ `push` ขึ้น GitHub แล้วเครื่องพัง = หายหมด (ทั้งที่ commit และไม่ commit)

### 3.4 rebase / merge / worktree ตอนมีงานค้าง
ดูข้อ 4 — operation พวกนี้เสี่ยงถ้ายังมีงานที่ยังไม่ commit ค้างอยู่
(เกี่ยวกับไฟล์ `soulbrew-rebase-worktree-th.md` ของคุณ — ระวังเรื่องนี้)

## 4. Switch Branch — เกิดเมื่อไหร่ และเกิดอะไรขึ้น?

### 4.1 branch สลับเมื่อไหร่?
```bash
git switch <branch>        # สลับไป branch อื่น
git checkout <branch>      # สลับ (คำสั่งเก่า)
git checkout -b <new>      # สร้าง branch ใหม่ + สลับไปเลย
git switch -c <new>        # เหมือนข้างบน (คำสั่งใหม่)
```
นอกจากนี้ branch/HEAD อาจขยับเองตอน: `git rebase`, `git merge`, `git pull`, หรือเครื่องมือ/agent บางตัวสั่งสลับให้

### 4.2 ถ้ามีงานที่ยังไม่ commit แล้วสลับ branch จะเกิดอะไรขึ้น?

**กรณีปกติ — git ปกป้องคุณ ✅**
- ถ้าไฟล์ที่คุณแก้ "ไม่ชนกัน" กับ branch ปลายทาง → git พางานที่ยังไม่ commit ติดไปด้วย (งานไม่หาย)
- ถ้าไฟล์ที่แก้ "จะถูกทับ" บน branch ปลายทาง → git **กั้นไว้ ไม่ยอมสลับ** แล้วขึ้น error:
  ```
  error: Your local changes to the following files would be overwritten by checkout.
  Please commit your changes or stash them before you switch branches.
  ```
  → แปลว่า git ไม่ให้สลับ เพื่อกันงานคุณหาย ต้อง `commit` หรือ `git stash` ก่อน

**กรณีอันตราย — บังคับสลับ ⚠️**
```bash
git checkout -f <branch>   # บังคับ → ทิ้งงานที่ยังไม่ commit
git switch --force <branch># บังคับ → ทิ้งงานที่ยังไม่ commit
```
อันนี้คือยอมทิ้งงานเพื่อสลับ ใช้เมื่อแน่ใจเท่านั้น

### 4.3 ทางเลือกที่ปลอดภัย: worktree
`git worktree` = แตกโฟลเดอร์ทำงานแยกอีกอันสำหรับอีก branch โดยไม่ต้องสลับใน branch เดิม → งานที่ค้างใน branch เดิมอยู่ครบ ไม่ต้อง commit ก่อน
(คุณมีโน้ต `soulbrew-rebase-worktree-th.md` อยู่แล้ว)

## 5. กฎง่ายๆ สำหรับคุณ

| สถานการณ์ | ทำอะไร |
|---|---|
| อยากกัน "เผลอ undo / กดทับงาน" | `git commit` |
| อยากกัน "เครื่อง/ดิสก์พัง" | `git commit` แล้ว `git push` |
| แค่พักงาน เลิกวันนี้ ไม่รันคำสั่ง git อันตราย | ไม่ต้องทำอะไร งานอยู่บนดิสก์แล้ว ✅ |
| จะสลับ branch แต่งานยังค้าง | `git commit` หรือ `git stash` ก่อน (หรือใช้ worktree) |
| จะ rebase / merge | commit งานค้างให้หมดก่อน แล้วค่อยทำ |
