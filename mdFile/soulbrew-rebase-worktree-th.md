# "Rebase on alpha" คืออะไร + worktree มีกี่ตัว (ฉบับภาษาไทย)

> สรุป ณ 2026-06-19 — อธิบายแบบเข้าใจง่าย 2 เรื่องที่งงบ่อยใน soulbrew team workflow:
> (1) คำสั่ง *"always rebase on alpha"* แปลว่าอะไร ทำไมต้องทำ
> (2) แต่ละ agent มี worktree แยกของตัวเองจริงไหม และจำนวน worktree = จำนวน agent หรือเปล่า
> ต่อยอดจาก `soulbrew-multi-agent-merge-th.md`
> ground: `maw-js/docs/codex-team-pattern.md`, `maw-js/CONTRIBUTING.md`

---

## 1. "Always rebase on alpha" คืออะไร

### `alpha` คืออะไรก่อน

`alpha` = **branch กลางอันเดียวที่งานทุกคน merge เข้าไปรวมกัน** — คิดว่าเป็น "โค้ดล่าสุดที่ตกลงกันแล้ว"
(ใน repo นี้: `main` = release เสถียรเท่านั้น, `alpha` = ที่ที่งานทุกอย่างลงจริง)

### rebase ทำอะไร — ดูทีละสเต็ป

agent B เริ่มงาน → แตก branch ออกจาก alpha:

```
alpha:   A1 ─ A2          ← B แตก branch ตรงนี้ ("alpha เวอร์ชัน 1")
                 \
B's branch:        B1 ─ B2   ← B commit ไป 2 อัน
```

ระหว่างที่ B เขียนอยู่ **agent A merge PR เข้าไปก่อน** → alpha ขยับไปข้างหน้า:

```
alpha:   A1 ─ A2 ─ A3 ─ A4   ← alpha ใหม่กว่าแล้ว ("เวอร์ชัน 2") มีงานของ A
                 \
B's branch:        B1 ─ B2     ← B ยังอิงของเก่า = "ตกขบวน / behind"
```

`git rebase alpha` สั่ง git ว่า:
*"เอา commit ของฉัน (B1, B2) พักไว้ก่อน → ย้ายจุดเริ่มมาที่ alpha ล่าสุด → แล้วเล่น commit ฉันทับลงไปใหม่"*
ผลลัพธ์:

```
alpha:   A1 ─ A2 ─ A3 ─ A4
                          \
B's branch:                B1' ─ B2'   ← งานของ B ไปนั่งบนโค้ดล่าสุดแล้ว
```

### เทียบให้เห็นภาพ

- **merge** = เอากระดาษของเราเย็บแทรกเข้าเอกสาร แล้วแปะ sticky note "รวมตรงนี้นะ"
- **rebase** = *เขียนงานของเราใหม่ทับบนเอกสารเวอร์ชันล่าสุด* ให้เหมือนเราเขียนทีหลังสุด → ประวัติเป็นเส้นตรง สะอาด

### แล้ว "always rebase on alpha" แปลว่าอะไร

= **กฎประจำตัวของ builder ทุกคน**: ก่อนเริ่มงาน และก่อนเปิด/อัปเดต PR ให้ **rebase branch ตัวเองบน alpha ล่าสุดเสมอ**
ได้ประโยชน์ 2 อย่าง:

1. **ไม่เคยสร้างงานบนโค้ดเก่า** — เริ่มจาก state ล่าสุดที่ merge แล้วทุกครั้ง
2. **ถ้าโค้ดเราชนกับของที่ merge ไปแล้ว → conflict โผล่ใน worktree ของเราเอง และเราเป็นคนแก้** —
   ไม่ลามไปที่ branch กลางให้ block คนอื่น (นี่คือจุดที่ agent "ตกขบวน" ดูดซับ conflict ไปแก้เอง)

## 2. worktree มีกี่ตัว — 1 ตัวต่อ 1 agent จริงไหม?

**เกือบใช่ — แต่มีข้อยกเว้น 1 อย่าง** คือ **1 worktree ต่อ 1 agent ที่ *เขียนโค้ด*** ไม่ใช่ต่อ *ทุก* agent

| agent                                             | `worktree` | ได้ worktree ไหม                                                                                                                           |
| ------------------------------------------------- | ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| **builder** (ตัวเขียนโค้ด)      | `true`     | ✅ มี**worktree แยกของตัวเอง** บน branch ตัวเอง — 1 builder = 1 worktree = 1 branch = 1 issue (ณ ขณะหนึ่ง) |
| **lead** (ตัวออร์เคสเตรชัน) | `false`    | ❌**ไม่มี** — เพราะ lead ไม่เขียนโค้ด ทำแค่แจกงาน/review/merge นั่งใน checkout หลักของ repo |

### สรุป

**จำนวน worktree ที่ active = จำนวน builder agent** (lead เป็นข้อยกเว้น ไม่มี worktree)

### ข้อสังเกตเพิ่ม

- builder มี **worktree active ทีละ 1 ตัว** เท่านั้น ณ ขณะหนึ่ง
- แต่ *เมื่อเวลาผ่านไป* ตัวเดียวอาจใช้หลาย worktree ได้ — ตอนถูกสั่งไป issue ใหม่ (`team reassign`)
  worktree เก่าจะถูกเคลียร์ แล้วสร้างอันใหม่ `fresh: true` ให้ issue ใหม่
- เลยเป็น 1:1 *ณ ขณะหนึ่ง* แม้ agent ตัวเดียวจะผ่านหลาย worktree ตลอดทั้ง sprint

## 3. ทำไม soulbrew ใช้ rebase ไม่ใช้ merge (ตอน merge โค้ด agent)

> เพิ่มเมื่อ 2026-06-21 — ตอบคำถาม "ทำไมใช้ git rebase แทน git merge เวลารวมโค้ดของ agent"

### คำตอบบรรทัดเดียว

soulbrew บังคับให้ builder แต่ละตัว **rebase branch ตัวเองบน `alpha`** แทนที่จะ merge `alpha` เข้า branch
เพราะ rebase **โยนภาระแก้ conflict ไปที่ agent ที่ตกขบวน ใน worktree ของตัวเอง** →
conflict ไม่เคยตกลงบน `alpha` (branch กลาง) และไม่ block agent ตัวอื่น

นี่คือหัวใจ — เหตุผลอื่นเป็นของแถม

### เหตุผล 4 ข้อ ที่ rebase ชนะ

**1. เจ้าของ conflict (เหตุผลจริง) 🎯**
หลาย agent ทำงานขนานกัน เมื่อ PR ของ A merge เข้า `alpha` ก่อน → B "ตกขบวน"

- ถ้า B **merge `alpha` เข้า branch** → ความยุ่งของ integration ลามเข้าใกล้ branch กลางได้
- ถ้า B **rebase บน `alpha`** → git เล่น commit ของ B ทับบน alpha ใหม่ หยุดที่บรรทัดที่ชน
  แล้ว **B แก้เองใน worktree ของ B** → `alpha` สะอาดเสมอ คนที่ทำให้ต่างคือคนที่แก้

> (อ้าง `soulbrew-multi-agent-merge-th.md`: *"conflict ไม่ได้ถูกกัน แต่ถูกโยนให้ agent ที่ตกขบวนไปแก้ตอน rebase"*)

**2. ประวัติเป็นเส้นตรง (linear history)**
N agent ขนานกัน — `merge` สร้างกราฟ merge commit รกเป็นใยแมงมุม, `rebase` ได้เส้นตรง:

```
merge:                      rebase:
alpha ─┬─ A ─┬─ M ─┬─ M     alpha ─ A ─ B ─ C   (เส้นเดียวสะอาด)
       └─ B ─┘     │
       └─ C ───────┘
```

→ `git bisect`, `git blame`, revert PR ทีละอัน ทำได้ง่ายและเดาได้

**3. PR สะอาด รีวิวง่าย**
PR ที่ rebase แล้วโชว์ **เฉพาะการแก้ของ agent ตัวนั้น เทียบกับ `alpha` ล่าสุด** ไม่มีอย่างอื่นปน
ถ้า merge แทน → branch/PR จะมี commit ของ alpha + merge commit ปนเข้ามา ทำให้ lead รีวิวยาก

**4. ทุกการ merge เข้า `alpha` ลงสะอาด (มักเป็น fast-forward)**
เพราะทุก branch rebase บน `alpha` ล่าสุดก่อน merge → ตอน lead merge ทีละอัน แต่ละอันลงโดยไม่ชน
ไม่มี conflict ที่จุดรวม ไม่มี merge commit รกบน `alpha`

### ข้อสำคัญ: ไม่ใช่ "rebase แทน merge" ทุกที่ — แต่แบ่งหน้าที่

| ใคร                               | ทำอะไร                                                        | เครื่องมือ            |
| ------------------------------------ | ------------------------------------------------------------------- | ------------------------------- |
| **builder** (แต่ละ agent) | อัปเดต branch ให้ทันสมัย + ดูดซับ conflict    | **rebase** บน `alpha` |
| **lead** (ตัวเดียว)    | รวม PR ที่เสร็จแล้วเข้า `alpha` ทีละอัน | merge จริง                  |

rebase = งานของ builder (อยู่กับโค้ดล่าสุด + เป็นเจ้าของ conflict ตัวเอง);
lead ยังเป็นคน merge รวมจริง — และมันลงสะอาด **เพราะ** builder rebase มาก่อน

### ข้อแลกเปลี่ยน (พูดตามตรง)

rebase **เขียนประวัติใหม่** → ต้อง `force-push` ปกติอันตราย แต่ปลอดภัยที่นี่เพราะ
**1 branch มีเจ้าของคนเดียว** (1 agent = 1 branch = 1 worktree) ไม่มีใครต่อยอดบน branch ของ B
การเขียนใหม่จึงไม่กระทบใคร — ถ้าเป็น branch *ที่ใช้ร่วมกัน* ห้าม rebase แบบนี้เด็ดขาด

## ตารางสรุปรวม

| คำถาม                               | คำตอบสั้น                                                                                                                       |
| ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `alpha` คืออะไร                 | branch กลางอันเดียวที่งานทุกคน merge เข้ารวมกัน (โค้ดล่าสุดที่ตกลงแล้ว)            |
| rebase ทำอะไร                      | ย้ายงานเราไปนั่งบน alpha ล่าสุด แล้วเล่น commit ทับใหม่ → ประวัติเส้นตรง           |
| ทำไม "always rebase on alpha"        | (1) ไม่อิงโค้ดเก่า (2) conflict โผล่ใน worktree เราเอง เราแก้เอง ไม่ block คนอื่น            |
| ทำไมใช้ rebase ไม่ใช้ merge | conflict ตกที่ agent ตกขบวน (ไม่ลง alpha) + history เส้นตรง + PR สะอาด + merge เข้า alpha ลงสะอาด |
| ใครใช้ rebase / ใคร merge       | builder = rebase; lead = merge รวมเข้า alpha ทีละอัน                                                                       |
| rebase ปลอดภัยเพราะอะไร  | 1 branch เจ้าของคนเดียว → force-push ไม่กระทบใคร                                                               |
| 1 worktree ต่อ 1 agent ไหม         | ใช่สำหรับ**builder**; **lead ไม่มี** worktree                                                                  |
| จำนวน worktree active               | = จำนวน builder agent                                                                                                               |
| 1 agent ใช้ได้กี่ worktree      | active ทีละ 1; ผ่านได้หลายตัวตามจำนวน issue ที่ทำ                                                         |
