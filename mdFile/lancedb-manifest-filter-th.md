# `.manifest` กรองจำนวนไฟล์ `.lance` ที่ต้องอ่านยังไง (ฉบับภาษาไทย)

> สร้างเมื่อ 2026-06-18 — อธิบายว่าเมื่อ Oracle ค้น vector ใน LanceDB จะต้องอ่านไฟล์ `.lance` กี่ไฟล์
> และทำไม **ไม่ได้อ่านทุกไฟล์** — `.manifest` เป็นตัว "กรอง" ว่าจะอ่านไฟล์ไหนบ้าง
> ground จากของจริงใน `~/.oracle/lancedb/oracle_knowledge.lance/`

---

## ✅ ตอบสั้นๆ ก่อน

ในโฟลเดอร์ `data/` มีไฟล์ `.lance` ทั้งหมด **19 ไฟล์** — แต่เวลาค้นจริง **ไม่ได้อ่านทั้ง 19 ไฟล์**

`.manifest` ของ version ล่าสุดจะบอกว่า "version นี้ใช้ fragment ไหนบ้าง" → ของจริงตอนนี้ชี้ไปแค่ **~3 ไฟล์**
อีก ~16 ไฟล์เป็น **ของเวอร์ชันเก่า (dead/history)** → ถูกข้าม ไม่อ่าน

➡️ **จำนวนไฟล์ที่อ่าน = จำนวน fragment ที่อยู่ใน manifest ล่าสุด ไม่ใช่จำนวนไฟล์ทั้งหมดบนดิสก์**

---

## 📁 โครงสร้างจริงของ dataset

```
~/.oracle/lancedb/oracle_knowledge.lance/
├── data/*.lance        ← ข้อมูลจริง (vector + text) — มี 19 ไฟล์บนดิสก์
├── _versions/*.manifest ← "สารบัญ" ของแต่ละ version (บอกว่า version ไหนใช้ไฟล์ไหน)
├── _transactions/*.txn  ← log การเขียนแต่ละครั้ง
└── _indices/            ← ❌ ไม่มี → ไม่มี ANN index → ใช้ brute-force (flat scan)
```

### ทำไมถึงมีหลายไฟล์ (ไม่เก็บไฟล์เดียว)
ไฟล์ `.lance` เป็นแบบ **เขียนครั้งเดียวแล้วแก้ไม่ได้ (immutable)**
เวลาเพิ่ม document ใหม่ → LanceDB **เขียนไฟล์ใหม่เล็กๆ 1 ไฟล์** ไม่ได้ไปแก้ไฟล์เดิม
เขียน ~19 ครั้ง → ได้ ~19 ไฟล์ (แต่ละไฟล์ = การเขียน 1 batch)

> ถ้าเก็บเป็นไฟล์เดียว → ทุกครั้งที่เพิ่มข้อมูลต้องเขียนทับทั้งไฟล์ใหม่ทั้งหมด → ช้า + เสี่ยงพังถ้าเขียนค้าง
> การแยกไฟล์ทำให้ "เพิ่มข้อมูล = แค่เขียนไฟล์ใหม่" ของเก่าไม่ถูกแตะ ปลอดภัยกว่า

## 🔍 ข้างใน `.manifest` ล่าสุดมีอะไร

ไฟล์: `_versions/18446744073709551595.manifest` (ขนาดแค่ **728 bytes** — เป็น binary/protobuf ไม่ใช่ text)

| ส่วน | เก็บอะไร |
|------|----------|
| **Schema** | คอลัมน์ของแต่ละแถว: `id` (string), `text` (string), `metadata` (string), `vector` (`fixed_size_list:float:768` — vector ขนาด **768 มิติ**) |
| **รายชื่อ fragment ที่ live** | ชื่อไฟล์ `.lance` ที่ version นี้ใช้ → **~3 ไฟล์** (ไม่ใช่ทั้ง 19) |
| **Transaction ref** | `19-81a35973-…​.txn` → การเขียนครั้งที่ 19 ที่สร้าง version นี้ |
| **Format version** | `lance 4.0.0` (รูปแบบ manifest) + `lance 2.0` (รูปแบบเก็บข้อมูล) |
| **Auto-cleanup config** | `older_than = 14days`, `interval = 20` → ลบ version เก่ากว่า 14 วันอัตโนมัติ ทุกๆ 20 version |

> 💡 manifest **ไม่เก็บข้อมูล/vector จริง** — เก็บแค่ metadata ที่ชี้ว่าให้ไปอ่าน fragment ไหน
> เลยมีขนาดแค่ 728 bytes ขณะที่โฟลเดอร์ `data/` ใหญ่ ~1.5 MB
>
> 💡 เพราะมี auto-cleanup → ไฟล์เก่าที่ตายแล้ว **ไม่ได้ค้างถาวร** จะถูกเก็บกวาดอัตโนมัติ

## 🔄 ขั้นตอนการค้น (ตรงนี้แหละที่ manifest กรองไฟล์)

```
ค้น vector: oracle_search "ข้อความ"
   │
   1. แปลง query เป็น vector (768 มิติ)
   │
   2. เปิด _versions/<ล่าสุด>.manifest   ← อ่านสารบัญก่อน
   │     → manifest บอก: "version นี้ = fragment A, B, C" (~3 ไฟล์)
   │
   3. อ่านเฉพาะ 3 ไฟล์นั้น                ← 👈 manifest กรองที่ "ระดับไฟล์" แล้ว
   │     (19 ไฟล์บนดิสก์ - ไฟล์ที่ตายแล้ว = เหลือ ~3 ไฟล์ที่ live)
   │
   4. สแกนเฉพาะคอลัมน์ vector ใน 3 ไฟล์นั้น → คำนวณ cosine distance → เรียงลำดับ
   │
   5. ดึงคอลัมน์ text เฉพาะ top-K ที่ชนะ (ไม่ดึง text ทุกแถว)
   │
   ▼
   ผลลัพธ์
```

### จุดสำคัญ: manifest กรอง "ระดับไฟล์" ไม่ใช่ "ระดับแถว"

| ระดับ | ใครเป็นตัวกรอง | กรองอะไร |
|-------|----------------|----------|
| **ไฟล์ / version** | **manifest** ✅ | เลือกว่า fragment ไฟล์ไหน live (อยู่ใน version ปัจจุบัน) |
| **แถว (row)** | `WHERE` clause (เช่น `project = 'x'`) | กรองว่าแถวไหนตรงเงื่อนไข ภายในไฟล์ที่อ่านมาแล้ว |

manifest แค่บอกว่า "ไฟล์พวกนี้คือ version ปัจจุบัน อ่านแค่นี้พอ" — ไม่ได้ดูเนื้อหาในแถว

## 🆚 ทำไมต่างจาก `oracle.db`

| | `oracle.db` (SQLite) | LanceDB |
|---|---|---|
| จำนวนไฟล์ | ไฟล์เดียว แก้ในที่ (in-place) | หลายไฟล์ immutable + manifest |
| ตอนเขียน | แก้ไฟล์เดิม (B-tree + WAL) | เขียนไฟล์ใหม่ ไม่แตะของเก่า |
| Versioning | ไม่มี (เก็บแค่สถานะปัจจุบัน) | เก็บทุก version → ย้อนกลับได้ (time-travel) |
| รวมเป็นไฟล์เดียว? | เป็นไฟล์เดียวอยู่แล้ว | ทำได้ด้วย `table.optimize()` (compaction) |

## 🧠 สรุปสั้น

- `data/` มี 19 ไฟล์ แต่ **ไม่ได้อ่านทั้งหมด**
- `.manifest` ของ version ล่าสุด = สารบัญ บอกว่า fragment ไหน live (~3 ไฟล์)
- ค้น = อ่านเฉพาะ fragment ที่ manifest ชี้ → สแกนคอลัมน์ vector → ดึง text เฉพาะ top-K
- manifest กรองที่ **ระดับไฟล์** (เลือก fragment) ไม่ใช่ระดับแถว
- ไฟล์เก่าที่ตายแล้วถูก auto-clean (เก่ากว่า 14 วัน) ไม่ค้างถาวร
- ถ้าอยากรวมไฟล์ให้เหลือน้อยลง/ไฟล์เดียว → ใช้ `table.optimize()`
