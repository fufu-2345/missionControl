# อธิบายโฟลเดอร์ `~/.oracle/` (ฉบับภาษาไทย)

> สรุป ณ วันที่ 2026-06-17 — อธิบายไฟล์ทุกตัวในโฟลเดอร์ `/home/chillox-intern/.oracle/`
> (ยกเว้นไดเรกทอรี `lancedb/` ตามที่ขอ)

## ภาพรวม

โฟลเดอร์นี้คือ **"สมองความรู้" ของ Oracle ทั้งก้อน** — เป็นระบบ `ψ→DB` ที่ memory note พูดถึง
กล่าวคือ vault `ψ` (ไฟล์ markdown ของเรา) จะถูก index เข้ามาเก็บเป็นฐานข้อมูลที่นี่

ตอน export ออกมายืนยันว่ามี **เอกสารทั้งหมด 55 รายการ** ดึงมาจาก `ψ/memory/...` แบ่งเป็น:

- `learning: 21` รายการ (จาก `ψ/memory/learnings`)
- `retro: 20` รายการ (จาก `ψ/memory/retrospectives`)
- `principle: 14` รายการ (จาก `ψ/memory/resonance`)

## 1. ฐานข้อมูลตัวจริง (กลุ่ม SQLite "สามพี่น้อง")

ไฟล์สามตัวนี้มาด้วยกันเสมอ — เป็น SQLite มาตรฐานที่ทำงานในโหมด **WAL (Write-Ahead Logging)**

| ไฟล์ | ขนาด | คืออะไร |
|---|---|---|
| `oracle.db` | 602 KB | **ฐานข้อมูลตัวจริง** ยืนยันว่าเป็น `SQLite 3.x` เก็บเอกสารที่ index ไว้ทั้ง 55 รายการ — คอลัมน์: `id, type, source_file, concepts, project, content` |
| `oracle.db-wal` | 0 B | **Write-Ahead Log** ข้อมูลที่เขียนใหม่จะลงที่นี่ก่อน แล้วค่อย "checkpoint" เข้าไฟล์ `.db` หลัก ตอนนี้ **ว่างเปล่า** แปลว่าทุกอย่าง flush เรียบร้อยแล้ว ไม่มีงานค้าง |
| `oracle.db-shm` | 32 KB | **Shared memory** สำหรับ WAL ช่วยให้หลาย connection ทำงานประสานกันได้ |

ไฟล์ `-wal` กับ `-shm` นั้น **SQLite จัดการให้อัตโนมัติ** เราไม่ต้องไปแตะ
ถ้าจะ copy ฐานข้อมูล ต้อง copy ครบทั้งสามตัว (หรือ checkpoint ก่อน) ไม่งั้นข้อมูลที่เพิ่งเขียนอาจหาย
การที่ `-wal` มีขนาด 0 byte = อยู่ในสถานะ "ปลอดภัยที่จะ copy ได้แล้ว"

## 2. ไฟล์สำรอง (Backup — สำเนาฐานข้อมูล ณ จุดเวลาหนึ่ง)

| ไฟล์ | ขนาด | คืออะไร |
|---|---|---|
| `oracle.db.premanual-2026-06-16.bak` | 401 KB | สำเนาที่ถ่ายไว้ **ก่อนทำงานแบบ manual** (ป้ายชื่อ `premanual`) ขนาดเล็กกว่า = เป็นสถานะ *ก่อนหน้า* ก่อนที่การ reindex ของวันนั้นจะทำให้ DB โตขึ้น |
| `oracle.db.backup-2026-06-16T07-19-36...` | 602 KB | สำเนาอัตโนมัติ ติด timestamp ละเอียดถึงวินาที |
| `oracle.db.backup-...07-20-53...` | 602 KB | อีกตัว ห่างกันราว 1.5 นาที |
| `oracle.db.backup-...07-20-56...` | 602 KB | อีกตัว ห่างกันราว 3 วินาที |

ทั้งหมดนี้คือสำเนาเต็มของ `oracle.db` ที่ถูกสร้างอัตโนมัติรอบ ๆ การทำงานที่เสี่ยง
(memory note บอกว่า `ψ→DB เป็น manual + 2 ขั้นตอน` — backup พวกนี้คือตัวคร่อมขั้นตอนนั้น)
timestamp ที่กระจุกกันแสดงว่ามีการ reindex ติด ๆ กันหลายรอบในวันที่ 16 มิ.ย.
ถ้าอยากได้พื้นที่คืน ลบตัวเก่า ๆ ออกได้

## 3. ไฟล์ส่งออก (Export — ข้อมูลในรูปแบบที่อ่านง่าย พกพาได้)

มี 3 timestamp แต่ละชุดมีทั้ง **`.json` และ `.csv`** — ข้อมูลเดียวกัน แต่คนละฟอร์แมต

| ฟอร์แมต | ข้างในมีอะไร |
|---|---|
| `*.export-*.json` (53 KB) | `{ "exported_at": ..., "count": 55, "documents": [ {id, type, source_file, concepts, content}, ... ] }` |
| `*.export-*.csv` (43 KB) | ข้อมูลแถวเดียวกันในรูปตาราง: คอลัมน์ `id, type, source_file, concepts, project, content` |

**ต่างจาก backup อย่างไร:** *backup* คือไฟล์ DB ตัวจริงแบบ binary (เอาไว้ restore)
ส่วน *export* คือ *เนื้อหา* ในรูปแบบเปิด — เอาไว้อ่าน, grep, เปิดใน spreadsheet หรือย้ายไปเครื่องมืออื่น
JSON เก็บโครงสร้างไว้ครบ (เช่น array ของ `concepts`) ส่วน CSV ทำให้แบนราบเพื่อเปิดใน Excel/pandas
